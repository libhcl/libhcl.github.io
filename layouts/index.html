libhcl

HCL parsers written in hand-written, dependency-free C — c-hcl, a reusable parser for the declarative HCL subset, and c-hcl2, a from-scratch full HCL2 implementation. No flex/bison, no third-party libraries — just cc *.c.

C99 zero dependencies no flex / no bison immutable AST fuzzed · allocation fault-injection BSD-3-Clause
Documentation GitHub

Two HCL parsers live here, both in plain C and both designed to be embedded as a git submodule. c-hcl covers the declarative subset of HCL native syntax — attributes and labeled blocks with scalar/list values — over a generic, zero-dependency AST (it was extracted from socket_vmnet). c-hcl2 is a from-scratch HCL2 implementation whose first milestone is the expression engine. Different scope, same place in the stack, same engineering bar.

Hand-written, not generated. The reference hashicorp/hcl pairs a ragel-generated scanner with a hand-written parser; libhcl is hand-written end to end for the same reasons — zero build dependencies (cc *.c is the whole build), precise line/column errors, and every line reachable by tests, including out-of-memory paths via an allocation-budget fault-injection hook. Each repo carries a GRAMMAR.md derived directly from its lexer/parser, ships a deterministic fuzzer, and stays clean under AddressSanitizer. BSD-3-Clause throughout.

Every library is hand-written C99 with no third-party code: parse once into an immutable AST, walk it with accessors, then free. Tests drive an allocation-budget fault-injection hook to reach out-of-memory paths, a deterministic fuzzer (make fuzz) shakes the parsers, and both build clean under AddressSanitizer. c-hcl sits at ~99% line coverage; c-hcl2 is milestone-based and not yet spec-complete. BSD-3-Clause throughout.
